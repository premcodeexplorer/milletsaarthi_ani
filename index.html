🎯 Step 1 — Orchestrator Initializes

The Orchestrator is the "boss". It creates an empty shared state dictionary and starts a loop (max 12 iterations). Each iteration, it asks the Planner what to do next.

state = {
  "image_path": "temp/grain.jpg",
  "location_text": "Baramati, Maharashtra",
  "quantity_quintal": 10.0,
  "moisture": 12.0,
  "year": 2026, "month": 4,
  "pipeline": [], "agentic_trace": []
}
Orchestrator asks Planner: "What should I run next?"

🧭 Planner — The "Brain"

The Planner inspects the current state and decides:

  • Which agents to run next
  • In parallel or sequential
  • Why (logged in agentic_trace for explainability)

Planner says: "State is empty. Run quality + geocode in parallel — they are independent."

⚡ Step 2 — Iteration 0: Quality + Geocode (PARALLEL)

Two independent tasks run simultaneously using Python's ThreadPoolExecutor. This saves time — instead of 50ms + 80ms = 130ms, total wall time = max(50, 80) = 80ms.

ThreadPoolExecutor — 2 threads running in parallel

🔍 Agent 1 — Quality Assessment

MobileNetV3-Large CNN

1. Opens grain image (224x224 resize)
2. Checks if image is blurry (Laplacian variance)
3. Runs CNN inference → softmax probabilities
4. Extracts millet type + grade from top class

Input: image_path
Output: millet: bajra grade: A confidence: 98.1%

📍 Geocode Tool

Nominatim OSM API

1. Sends location text to Nominatim API
2. Gets latitude, longitude, district, state
3. Resolves "Baramati, Maharashtra" to exact coordinates

Input: location_text
Output: lat: 18.22 lon: 74.45 district: Pune

Both results merged into shared state dict

🚦 Step 3 — Planner Gates (Conditional Routing)

After Quality runs, the Planner checks multiple conditions before proceeding. This is dynamic routing — the pipeline changes based on results.

Gate 1: Blurry Image?
Condition: Blur score < 500
Action: Pipeline ABORTS
Output: "Image is too blurry. Please retake."
No agents run after this.
Gate 2: Very Low Confidence?
Condition: quality_score < 30%
Action: Route to HUMAN_REVIEW
Output: "Cannot identify grain. Retake photo."
Pipeline short-circuits to human review.
Gate 3: Weak Confidence?
Condition: 30% ≤ score < 55%
Action: REFLECTION LOOP
Process:
1. Enhance image (contrast +40%, sharpness +30%, brightness +10%)
2. Re-run Agent 1 on enhanced image
3. Keep better result
Max 1 retry allowed.
Gate 4: Good Confidence
Condition: score ≥ 55%
Action: CONTINUE to next step
Status: Normal path
This is the happy path. Most images pass this.

🔄 Reflection Loop Detail (Self-Correction)

This is an agentic self-correction pattern. The system recognizes its own uncertainty and tries to improve:

Agent 1 says: "I'm only 42% sure this is Jowar Grade B"
  ↓ Planner triggers reflection
Orchestrator enhances the image (PIL ImageEnhance)
  ↓
Agent 1 re-runs on enhanced image → "Now 68% sure it's Jowar Grade A"
  ↓ Compare: 68% > 42% → Use new result
    (If worse → keep original)

💧 Urgency Gate: High Moisture

If moisture > 16%, the Planner takes the urgent path:

  • Skips price trend analysis (no time to wait)
  • Runs market + weather immediately
  • Decision agent runs in URGENT mode
  • Likely output: URGENT_SELL

⚡ Step 4 — Iteration 1: Market + Weather (PARALLEL)

Both agents only need lat/lon (from geocode), so they run simultaneously.

ThreadPoolExecutor — 2 threads running in parallel

🗺️ Agent 3 — Market Comparison

XGBoost + OSRM Routing

1. Finds APMCs within 300km (haversine filter)
2. Gets real road distance via OSRM API
3. Calls Agent 2 for price at each APMC
4. Calculates: transport = distance x ₹30/km / quantity
5. Commission: 1.5% (MH) or 2% (KA)
6. Net = APMC price - transport - commission
7. Ranks nearest 5, picks best by net price

Input: lat/lon millet grade
Output: 5 nearest APMCs best_market net_price MSP check

🌦️ Weather Tool

Open-Meteo API (Free)

1. Fetches 7-day forecast for farmer's location
2. Extracts: rainfall, rain probability, max temperature
3. Generates summary:
  • Heavy rain likely? → Spoilage warning
  • Hot week (≥38°C)? → Storage risk
  • Clear and dry? → Safe to wait

Input: lat lon
Output: rain_mm_3d hot_days summary

🛡️ Degraded Mode — Error Handling

If any agent crashes, the pipeline doesn't break. The Orchestrator wraps every call in _safe_call():

  • Agent returns error → logged in pipeline[]
  • Other agents still continue
  • If market fails, weather still runs (and vice versa)
  • Planner checks and retries if needed

💰 Step 5 — Iteration 2: Price Prediction (SEQUENTIAL)

Runs after Market Agent because it needs the district from Agent 3 to predict the right regional price. This is why it can't run in parallel with Market.

💰 Agent 2 — Price Prediction

XGBoost Regressor

1. Takes millet type, grade, district, state, month
2. Encodes categorical features (LabelEncoder)
3. Runs XGBoost model → predicted ₹/quintal
4. Compares current vs last month → trend (rising/falling/stable)
5. Calculates ±5% price range
6. Total revenue = price x grade_multiplier x quantity

Input: millet grade district month
Output: ₹880/quintal trend: rising total: ₹8,805

📊 Grade Multiplier

GradeMultiplierMeaning
A1.10x<2% broken, <1% foreign
B1.00x2-5% broken, 1-3% foreign
C0.88x>5% broken, >3% foreign

Why sequential? Agent 2 needs district from Agent 3's geocoding to stay in-distribution. Running it without the correct district would give wrong regional prices.

Why ThreadPoolExecutor (not AsyncIO)?

All agents are I/O-bound (API calls to OSRM, Nominatim, Open-Meteo) or GIL-releasing (PyTorch inference releases GIL during C++ tensor ops). Threads give real parallelism for these workloads without needing async/await complexity.
